# Tuning campaign 2026-07-02 — what was tuned, what was measured, what was rejected

One overnight campaign to push the serving recipe past the documented ~13 tok/s.
Outcome: **15.3 tok/s single-stream / 28.9 tok/s aggregate @ C4**, plus two
infrastructure bug-fixes that invalidate some earlier measurements (in a good way).

## Final standing config (the tuned recipe)

```bash
vllm serve <QuantTrio/GLM-5.2-Int4-Int8Mix> \
  --tensor-parallel-size 4 --distributed-executor-backend mp --nnodes 4 \
  --kv-cache-dtype fp8 --max-model-len 65536 \
  --max-num-seqs 4 \                       # was 2 — enables C4 aggregate
  --max-num-batched-tokens 2048 \          # prefill-chunk size, see RAM note
  --gpu-memory-utilization 0.86 \          # hard OOM-safety cap on GB10
  --num-gpu-blocks-override 1152 \         # MANDATORY, see estimator note
  --enforce-eager \                        # yes, really — see compile note
  --speculative-config '{"method":"mtp","num_speculative_tokens":3}' \
  --reasoning-parser glm45 --enable-auto-tool-choice --tool-call-parser glm47
# env: VLLM_SPARSE_INDEXER_MAX_LOGITS_MB=256  (long-prompt RAM cap)
```

## Results (true-token accounting, see bench note)

| Test | Result |
|---|---|
| Single-stream decode | **15.3 tok/s** (14.1–15.7 across runs) |
| C2 aggregate | **19.8 tok/s** |
| C4 aggregate | **28.9–29.4 tok/s** |
| Decode @ 18K-token context depth | **13.7 tok/s** |
| TTFT @ 18K-token prompt | 67 s (~270 tok/s prefill — the weak axis) |
| KV pool | 73,728 tokens (fp8, shared across streams) |
| RAM headroom, worst case (18K prefill) | head 2G / workers 3–5G |
| Quality | unchanged weights — HumanEval 96.3 / GSM8K 97.5 evals stand |

## Finding 1 — the bench undercounted: "13 tok/s" was never the real speed

The original bench counted **SSE stream chunks**, but with MTP speculative decoding
vLLM delivers **2–3 accepted tokens per chunk**. All chunk-counted speeds are
undercounts. The corrected bench (`benchmarks/bench-true-tokens.sh`) uses
`stream_options: {"include_usage": true}` and reads `usage.completion_tokens`.
All numbers in this doc are true tokens.

## Finding 2 — CUDA graphs / torch.compile are a NET LOSS for GLM-5.2 on sm_121a

Tested three ways on the rebuilt 0.23 b12x image: default VLLM_COMPILE piecewise+full
graphs, capped capture sizes, and `custom_ops: ["all"]` (hand-written CUDA ops inside
the compiled graph). All were **~2× slower than eager** (chunk-measured 6.3–6.6 vs 13).
MTP acceptance was healthy and Marlin MoE was engaged, so the overhead is the
compile/piecewise machinery itself across 78 DSA layers (`sparse_attn_indexer` is a
splitting op → per-layer graph boundaries). **`--enforce-eager` is the right mode on
this stack** — question closed. (The 8× win in vLLM-0.24 DSpark ports comes from
compiled *flashinfer attention kernels*, a different mechanism, and remains the
promising future track.)

## Finding 3 — vLLM's KV estimator computes NEGATIVE blocks on unified memory

At util 0.86 the estimator produced `num_gpu_blocks=-28` (it treats ambient
system/page-cache RAM on GB10's unified memory as consumed GPU memory).
`--num-gpu-blocks-override` is therefore **mandatory, not an optimization**.
1152 blocks = 73,728 tokens = 3.7 GiB fp8 KV; real per-node footprint ≈ 97 (weights)
+ 3.7 (KV) + activations ≈ 105–108 GiB, leaving proper OOM headroom on 121 GiB nodes.

## Finding 4 — the root-zombie gpu-clear bug (affects ANY crashed run)

Crashed vLLM runs leave **root-owned worker processes holding up to 106 GiB** of
unified RAM per node. A non-root `kill -9` fails **silently**, so cleanup scripts
"succeed" while the zombie stays — every subsequent launch then sees less free memory
(we watched available KV shrink -1.8 → -2.4 → 0.99 → 0.21 GiB across identical
configs before finding it). Fix: `sudo kill -9` in gpu-clear + verify with
`nvidia-smi --query-compute-apps` on **every** node before any launch.

## Finding 5 — long-prompt RAM spikes: two required caps

At an 18K-token prompt the DSA indexer + prefill activations drove available RAM to
0–1 GiB (crash territory). Two changes restored 2–5 GiB headroom at **zero TTFT cost**:
`VLLM_SPARSE_INDEXER_MAX_LOGITS_MB=256` and `--max-num-batched-tokens 2048`
(halved prefill chunks; the indexer cap absorbed the would-be TTFT regression).

## Tried and rejected (so you don't have to)

| Idea | Verdict |
|---|---|
| CUDA graphs / compile (3 variants) | 2× slower than eager (Finding 2) |
| W4A4 NVFP4 checkpoint via Marlin | 9.9 tok/s + garbage output + ~102 GiB/node (Marlin W4A16 expansion erases the FP4 byte win); native FP4 GEMM (cuteDSL/cutlass/trtllm) hardware-rejects sm_121a |
| MTP k=4/5 | per-position acceptance @k=3 is 0.96/0.87/0.58 (code), 0.70/0.42/0.24 (prose) — position 3 already marginal |
| util > 0.86 | violates GB10 OOM-safety envelope; also workers' preflight ceiling is ~0.898 (kernel carve-out) |
